# Developing change-review

Instructions for working **on this codebase** — written for coding agents, useful for humans too. (The instructions for *using* the tool live in [skills/change-review/SKILL.md](skills/change-review/SKILL.md): that folder is the skill, installed via `npx skills add marinsokol5/change-review` — which copies it into the agent's skills dir — with the CLI shipping inside it under `scripts/`.)

## What this is

A zero-runtime-dependency CLI that lets an AI agent request human review of a diff: the CLI serves a single-file diff UI on localhost, the human leaves line comments and a verdict in the browser, and the verdict comes back to the agent as JSON on stdout with a meaningful exit code. See [README.md](README.md) for the user-facing story. The CLI is TypeScript that `node` runs directly via native type stripping — there is no build step, no `dist/`, and nothing to install at runtime. (An earlier npm-package variant with a PreToolUse hook lives on the `hook-version` branch.)

## Setup

```bash
cd skills/change-review/scripts
npm install      # dev-only: TypeScript + @types/node, used exclusively for type-checking
npm run check    # tsc --noEmit over reviewer.ts + src/
npm run demo     # smoke test: opens examples/demo.patch in the browser
```

- Node >= 22.18 (native type stripping), ESM throughout, TypeScript as the only (dev) dependency.
- **No build step**: `node reviewer.ts …` runs the sources as-is, so changes take effect on the next command (a detached server keeps running old code until it exits — kill it or start a new session). `ui/index.html` is read from disk on every request, so UI changes just need a page reload.
- Everything the CLI writes goes under the per-invocation `--dir` (default `<os-tmpdir>/change-review`); pass a scratch `--dir` when testing and delete it afterwards.

## Repo layout

The skill is `skills/change-review/` (manifest `SKILL.md` + the CLI package in `scripts/`); `npx skills add` copies that folder verbatim. Paths below are relative to `skills/change-review/scripts/` unless noted.

| path | role |
|---|---|
| `reviewer.ts` | the CLI entrypoint agents run (`node <skill-dir>/scripts/reviewer.ts`) — arg parsing + all commands (`review`, `wait`, `answer`, `result`, `list`, `serve`, `config`); extracts the global `--dir` before dispatch; spawns the detached server; turns the result into stdout JSON + exit code, embedding its own absolute path and the data dir in every suggested follow-up command |
| `../SKILL.md` | the agent-facing contract (`skills/change-review/SKILL.md`); documents invocation, `--dir` discipline, exit codes, and the verdict JSON |
| `src/session.ts` | session store under `<--dir>/<id>/` (`setDataDir`/`dataDir`) — request.json, patch.diff, result.json, server.json, threads.json (Discuss threads), history/ per round (each round's patch + result, read back by `readHistoryPatch`/`readHistoryResult` for round browsing), proposal/ + proposal-base/ + apply.json (staged proposed AND base bytes + sha256 manifest, so approve — full or partial — applies deterministically), applied.patch/revert.patch (partial-approve artifacts); outcome polling (verdict or open discussion); reply/answer validation |
| `src/config.ts` | user config at `~/.change-review/config.json`; one key, `waitMode` (`"stop"` default \| `"poll"`), surfaced as the `config wait-mode` command — the only path the tool touches outside `--dir` |
| `src/server.ts` | localhost HTTP server — serves the UI (also at `/round/<n>` — one SPA, routed client-side), `GET /api/session[?round=N][&diffAgainst=M]` (parsed diff for that round + `rounds` nav list + threads; a historical round comes back `readOnly: true` with its archived verdict and comments; `diffAgainst` returns the interdiff between rounds M and N instead), `POST /api/submit` (validates — including optional `skipped` chunk refs on approve — attaches each discussed comment's thread as `comments[].discussion`, applies a staged proposal on approve filtered to the kept chunks, writes applied.patch/revert.patch + a `chunks` outcome on partial approves, writes result.json, exits ~500 ms later — a discussion never blocks it), `POST /api/questions`/`/api/answer`/`/api/close-thread` (Discuss threads; `/api/questions` returns the created thread ids so the UI links them to comments) |
| `src/patch.ts` | unified-diff parser (git and plain diffs, renames, binary, `\ No newline`, content lines that look like headers). Most edge-case-prone code in the repo; still untested |
| `src/chunks.ts` | per-chunk apply machinery: run (chunk) identity shared with the UI, selection validation, `filterFiles` (base → kept-chunks diff), `revertFiles` (fully-applied tree → kept-chunks diff), a unified-diff serializer `git apply` accepts, and `applyFileDiffToContent` — the exact-splice applier (no fuzzing) behind partial proposal applies |
| `src/interdiff.ts` | diff *between two rounds* of the same review (the ⇄ compare view): reconstructs each round's proposed file from its patch — base regions neither patch touches become placeholder tokens matched by base line number, valid because both rounds diff the same base — then Myers-diffs the reconstructions into the same `FileDiff[]` shape the UI renders. Unknown-content lines can only ever be unchanged context, so they're never displayed (they split hunks instead); a base disagreement between the rounds sets `FileDiff.warning` |
| `src/proposal.ts` | builds a patch from a proposal dir via `git diff --no-index`, relabelling temp paths to repo-relative ones; also returns the list of changed files so `cmdReview` can stage them for apply |
| `src/annotate.ts` | annotation mode (`review --file`): synthesizes a zero-change, all-context unified diff so the UI shows current file contents for line comments; the user's comments become the spec and the agent's edits arrive as round 2 |
| `src/open.ts` | best-effort cross-platform browser open (`CHANGE_REVIEW_NO_OPEN=1` suppresses it) |
| `ui/index.html` | the entire frontend in one self-contained file — no build step, no CDN, drafts (comments, summary, chunk selection) persisted in localStorage; per-chunk apply toggles live in the diff gutter and drive the morphing Apply button; keyboard shortcuts (`j`/`k` chunk nav anchored to the clicked line, `x` toggle, `c` comment, `d`/`r`/`⌘⏎` send actions, `?` help) with an Enter-to-confirm modal before every terminal action; light/dark theme follows the OS, and the header toggle's override persists in the `change_review_internals_theme` cookie — host-scoped, so unlike localStorage it survives the per-session ephemeral port (verbose name because every localhost app sees it) |

## How a review flows internally

1. `cmdReview` obtains a patch (from `git diff`, a proposal dir, a patch file, stdin — or synthesized all-context by `--file` annotation mode) and calls `session.createSession`. Reusing an id (`--session`) bumps `round`, archives the previous round's patch/result into `history/`, clears any staged proposal, and kills any stale server. With `--replies`, each agent reply is validated against the previous round's comments (one per comment, enforced) and merged into the archived result as `comments[].reply` — that's what the UI threads under each comment when browsing that round at `/round/<n>`. In proposal mode, `session.stageProposal` then snapshots the changed files' proposed bytes into `proposal/` — and their base bytes into `proposal-base/` — and writes `apply.json` (base + proposed sha256 per file), so the approve-time apply (full or partial) doesn't depend on the agent's staging dir still existing.
2. `ensureServer` health-checks `server.json`'s pid/port (`GET /api/health` must echo the session id); if dead, it spawns `node reviewer.ts serve <id> --dir <dataDir>` **detached** and waits for it to come up (the data dir is always passed explicitly, so the server never depends on the default resolving the same way).
3. The CLI opens the browser, then polls `result.json` *and* `threads.json` every 500 ms via `session.waitForOutcome`.
4. Instead of a verdict, the user can click **Discuss** to send their line comments to the agent for a reply-per-comment (`POST /api/questions` → `threads.json`; each comment becomes a thread and the endpoint returns the new thread ids, aligned to input order, so the UI links them back to the persisted local comments). The waiting CLI returns early with exit 5 and the open comments on stdout; the agent posts replies with the `answer` command (validated all-or-nothing, then `POST /api/answer`) and goes back to waiting. A discussion never blocks the verdict — `/api/submit` accepts a verdict whether or not a discussion is still open, and the UI keeps the terminal buttons live. In the UI a comment offers Edit / Delete only until it's discussed; once sent to the agent its body is locked (editing would desync a conversation the agent already answered), but the discussion itself is multi-turn: once the agent replies, a Reply box under the thread sends a follow-up (`POST /api/questions` with `{ thread, body }`), which reopens the thread — the waiting CLI returns exit 5 again with the exchange so far in `history`, and the agent answers with the same `answer` flow, as many turns as needed. On the terminal verdict, each discussed comment's thread messages are attached to it as `comments[].discussion`. (Comments and "Discuss" share one storage: `threads.json` — historically the "question threads" — with internal names unchanged.)
5. The UI submits to `/api/submit` — on Apply, together with the `skipped` chunk refs (`{file, hunk, run}` / `{file, binary}`) the user deselected. The server validates them against the round's parsed patch, and on an approve of a proposal session runs `session.applyProposal` — writing each file's target bytes into the session's `cwd`: the staged proposal when fully kept, the staged base when fully skipped, base-plus-kept-runs (exact line splice via `applyFileDiffToContent`) when partial. All-or-nothing: if any file's current contents match neither the reviewed base hash nor its target (it drifted mid-review), nothing is written and the conflicts are reported. The outcome lands in the result as `apply` (`{ applied, wrote, conflicts?, error? }`) plus, for selections, `chunks` (`{ total, applied, skipped[], appliedPatch?, revertPatch? }` — the two patch artifacts let worktree/patch-mode agents land the approved subset deterministically). Then it writes `result.json` atomically (tmp + rename) and exits shortly after.
6. The CLI prints the result and exits 0/2/3 — or, if its `--timeout` fired first, prints a pending JSON and exits 4. The detached server keeps running, so the review can be picked up later. What the pending JSON's `hint` tells the agent depends on the configured `wait-mode`: `stop` (default) = end the turn and let the user resume (the `result` command picks up the verdict — or the open questions), `poll` = keep re-running `wait`. Both hints embed the CLI's absolute path and the `--dir`, so they are copy-paste runnable.

## Invariants — don't break these

1. **stdout is machine-readable only.** Verdict/pending JSON is the sole stdout output; all human-facing messages go to stderr via `info()`/`fail()`. Agents parse stdout.
2. **Exit codes are API:** 0 approve, 2 request_changes, 3 reject, 4 pending, 5 questions for the agent, 1 error.
3. **The server must outlive the CLI.** Agents run commands with timeouts; the review survives because the server is detached and the verdict lives on disk. Never tie a verdict's fate to the spawning process.
4. **No runtime dependencies, single-file UI, binds 127.0.0.1 only.** The server also rejects requests with a non-local `Host` or `Origin` header (DNS-rebinding/CSRF defense — `rejectNonLocal` in `src/server.ts`); don't weaken it to make a client work, fix the client's headers instead.
5. **Runs directly under Node's type stripping — keep it that way.** No build step, ever: only erasable TypeScript syntax (no enums, namespaces, or parameter properties — `erasableSyntaxOnly` enforces it), type-only imports marked `import type` (`verbatimModuleSyntax` enforces it), and relative imports spelled with real `.ts` extensions. If `npm run check` passes but `node reviewer.ts` doesn't start, you broke this.
6. **`skills/change-review/SKILL.md` is part of the contract.** Changing flags, the JSON shape, or exit codes means updating `SKILL.md` and the README in the same change. Installed copies are copies of the `skills/change-review/` folder; users pick changes up with `npx skills update change-review`.
7. **Everything lives under `--dir`.** The agent chooses the data dir per review session; the same dir must reach every command (the detached server gets it explicitly on its command line, and every emitted hint echoes it). Never write to a fixed global path — the sole exception is `~/.change-review/config.json`, written only by an explicit `config` command.
8. **Approve on a proposal session applies exactly the reviewed bytes — filtered to the reviewer's chunk selection — and only the CLI writes them.** The server applies the staged snapshot (verified by sha256; partial targets are rebuilt from the staged base by exact splice) at verdict time; agents are told not to re-write the files. Keep it all-or-nothing on conflicts: a half-applied proposal is worse than a refused one.
9. **Chunk identity is shared UI ↔ server.** A chunk is (file, 0-based hunk index, 0-based run index) over `parseUnifiedDiff` output, where a run is a contiguous block of +/- lines; `hunkRuns` exists twice — `ui/index.html` and `src/chunks.ts` — and the two MUST derive runs identically. Change them together or selections silently target the wrong chunks.

## Testing changes

No automated tests yet (planned — see [ROADMAP.md](ROADMAP.md)). Manual loop:

- `npm run check` type-checks; `npm run demo` boots the committed 3-round demo review (`examples/demo-session/`, copied to a fresh temp dir per run so verdicts never dirty the repo): rounds 1–2 archived with threaded replies and a Discuss exchange, round 3 live with an answered thread — the standing fixture for UI work and README screenshots. `npm run demo:rebuild` regenerates the snapshot by replaying the story through the real CLI + server (`examples/build-demo.ts`, run unsandboxed) — do that whenever the session format changes. `npm run demo:fresh` opens `examples/demo.patch` as a new single-round review.
- `node reviewer.ts serve <id> --dir <dir>` runs the server in the foreground to see errors.
- `node reviewer.ts list --dir <dir>` shows that dir's sessions; use a scratch `--dir` and delete it when done.
- `--worktree` needs a git repo with uncommitted changes; `--proposal` and patch mode work anywhere.
- If you're an agent testing this from inside a sandbox: binding localhost is usually blocked (`listen EPERM`) — re-run unsandboxed.

## Gotchas

- Relative imports use **`.ts` extensions** in the sources (`./session.ts`, not `./session.js` — there is no compile step to rewrite them; `allowImportingTsExtensions` makes tsc accept it).
- `parseUnifiedDiff` counts hunk lines from the `@@` header so that content lines starting with `---`/`+++`/`diff ` aren't mistaken for file headers — keep that property when touching it. The `\ No newline at end of file` marker is handled *before* that count gate, because it can trail the hunk's final line after the counts are exhausted — dropping it silently breaks byte-exact partial applies.
- A failed health check does **not** kill the recorded pid (it may have been reused by another process); it just forgets `server.json` and starts fresh.
- `POST /api/submit` returns 409 only if a verdict already exists for the round — rounds are single-shot by design. An open Discuss thread does NOT block it (a discussion never blocks the verdict).
- **Only the server writes `threads.json`** — the CLI posts answers over HTTP so writes serialize through one event loop. Don't add direct CLI writes.
- Historical rounds are read-only by server contract, not just UI: `GET /api/session?round=N` serves the archived snapshot with `readOnly: true`, and out-of-range rounds 404. `/round/<n>` serves the same single-file SPA — the round comes from `location.pathname` client-side. `POST /api/submit` always targets the current round; don't wire it to a viewed historical round. The same goes for the compare view (`/round/N?diffAgainst=M`): always `readOnly: true`, no comments, no threads — it's a lens, not a reviewable surface.
- The interdiff (`src/interdiff.ts`) assumes both rounds were diffed against the same base. In worktree mode that can drift (a commit between rounds); it detects disagreements via base-line content conflicts and surfaces `FileDiff.warning` rather than refusing.
- Pure-context hunks (annotation mode) are valid diffs everywhere in the pipeline: they parse, render (without the `@@` header row), and produce **zero chunks** — the UI detects "all files all-context" and shows the annotation notice plus a "Looks good" Apply button. Don't add code that assumes a hunk contains at least one +/- line.
